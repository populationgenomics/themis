# Repository structure

A polyglot monorepo, grouped by role — each top-level directory has one job. The layout anticipates growth (the Next.js
web app, internal services, a sandbox worker) without standing up tooling for parts that do not exist yet.

| Path        | Holds                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `apps/`     | Deployable application surfaces, one directory per service, each owning its Dockerfile. `apps/web` is the IAP-fronted web surface — the Next.js (App Router) web application (UI + BFF, and the Managed-Agents client) on the Bun runtime. Future: `apps/sandbox-worker`.                                                                                                                                                                                                                                                                                                                                                                                                                     |
| `themis/`   | The internal Python code as one PEP 420 namespace tree (`themis.*`, a single `pythonpath` root): `themis.services.<name>` (data-plane gRPC services, each subclassing the servicer base generated from its hand-authored proto — see [`design/services.md`](design/services.md)), `themis.rpc.<domain>` (the generated protobuf messages + gRPC stubs), `themis.clients.<name>` (client-side helpers for calling a service), `themis.migrate` (the SQL migration runner), and at-rest domain libraries `themis.<domain>` (e.g. `themis.litcache` — a proto-native at-rest domain with no wire surface; see [`design/proto.md`](design/proto.md)). An image copies only the subtrees it needs. |
| `packages/` | Shared libraries imported by more than one app (shared types, API clients, config schemas), TypeScript and Python alike. Added when the first shared module appears.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| `schema/`   | The schema IDL: hand-authored `.proto` under `schema/proto/` — the source of truth for shapes across languages. Codegen is local (`grpcio-tools` protoc for the Python stubs, `protoc-gen-es` for the web tier's protobuf-es); `buf` gates it (`buf lint` + `buf breaking`). TypeSpec/Zod/JSON-Schema retired; enums are proto-canonical and declared-field constraints are protovalidate options. One source of truth (`docs/design/proto.md`).                                                                                                                                                                                                                                              |
| `infra/`    | Cloud infrastructure only (Pulumi, Python) — no application code. One stack per environment; images are built in `apps/*` by CI and deployed here.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| `docs/`     | Design docs, runbooks, style guides. Primary audience is a model.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| `tools/`    | Repo tooling (CI screens, scripts) — *not* the agent's tools, which are apps/services.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| `tests/`    | Cross-cutting tests; app-local tests live with their app.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |

As these pieces land, JS apps and packages go under a **Bun** workspace and Python under a **uv** workspace (dependency
groups).
