# Repository structure

A polyglot monorepo, grouped by role — each top-level directory has one job. The layout anticipates growth (the Next.js
web app, agent tool + MCP servers, a sandbox worker) without standing up tooling for parts that do not exist yet.

| Path        | Holds                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `apps/`     | Deployable application surfaces, one directory per service, each owning its Dockerfile. `apps/web` is the IAP-fronted web surface — the Next.js (App Router) web application (UI + BFF, and the Managed-Agents client) on the Bun runtime. Future: `apps/sandbox-worker`, `apps/<name>-mcp` (tool / MCP servers).                                                                                                                                                                                                             |
| `themis/`   | The internal Python code as one PEP 420 namespace tree (`themis.*`, a single `pythonpath` root): `themis.services.<name>` (data-plane gRPC services, each subclassing the servicer base generated from its TypeSpec-authored proto — see [`design/services.md`](design/services.md)), `themis.rpc.<domain>` (the generated protobuf messages + gRPC stubs), `themis.clients.<name>` (client-side helpers for calling a service), and `themis.migrate` (the SQL migration runner). An image copies only the subtrees it needs. |
| `packages/` | Shared libraries imported by more than one app (shared types, API clients, config schemas), TypeScript and Python alike. Added when the first shared module appears.                                                                                                                                                                                                                                                                                                                                                          |
| `schema/`   | The schema IDL: TypeSpec (`.tsp`) sources plus the emitters that produce the committed artifacts — JSON Schema + Pydantic for at-rest domains, Protobuf + gRPC stubs for wire services, plus Zod for a frontend consumer. One source of truth for shapes across languages (`docs/design/typespec.md`). Its own Bun toolchain.                                                                                                                                                                                                 |
| `infra/`    | Cloud infrastructure only (Pulumi, Python) — no application code. One stack per environment; images are built in `apps/*` by CI and deployed here.                                                                                                                                                                                                                                                                                                                                                                            |
| `docs/`     | Design docs, runbooks, style guides. Primary audience is a model.                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| `tools/`    | Repo tooling (CI screens, scripts) — *not* the agent's tools, which are apps/services.                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| `tests/`    | Cross-cutting tests; app-local tests live with their app.                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |

As these pieces land, JS apps and packages go under a **Bun** workspace and Python under a **uv** workspace (dependency
groups).
